# SEO + 安全设计增强版落地方案（massage-next）

> 用途：把“德国单店按摩店：SEO + 安全设计增强版提示词”沉淀为 `massage-next` 项目的长期执行基线。
> 
> 目标：避免后续开发继续依赖聊天上下文，直接按仓库文档推进，实现更快、更稳定地向“完整可运营后台 + 可上线官网”收口。

---

## 1. 项目定位（当前统一口径）

`massage-next` 是一个面向 **德国本地单店按摩店** 的：

- **双语官网**（德语 / 英语）
- **中文后台管理系统**
- **轻量、单管理员、单体优先** 的可运营网站系统

当前阶段统一目标：

1. 官网展示与本地 SEO 可用
2. 在线预约流程可用
3. 中文后台可运营
4. 安全能力从设计表达逐步落为真实能力
5. 部署与维护成本保持低复杂度

### 当前不做什么
- 多门店体系
- 重型 RBAC / 企业级权限系统
- ERP / CRM 大系统
- 在线支付
- 复杂会员系统
- 微服务拆分

### 明确保留的后续扩展位
- 功能开关（feature flags）
- WhatsApp 通知
- 优惠券 / 营销活动
- 会员系统
- 2FA
- 更正式的安全日志 / 审计能力

---

## 2. 当前开发总原则（以后持续执行）

### 2.1 产品原则
1. 单店优先
2. 单管理员优先
3. 预约闭环优先
4. 后台轻量优先
5. SEO 从架构层考虑，不是上线前补丁
6. 安全优先，但避免一开始过度企业化
7. 所有新增能力尽量支持逐步增强

### 2.2 开发原则
1. 优先高价值、低风险、可快速验证的能力
2. 优先补工作台能力，不先堆复杂概念
3. 能复用现有数据结构就先复用
4. 优先文档化，再持续实现
5. 每阶段固定执行：
   - 更新文档
   - `npm run build`
   - commit
   - push
   - 汇报

### 2.3 后台产品模式（固定）
后台统一按以下模式继续演进：

> **先看状态 → 再找入口 → 再处理 → 再追踪结果**

该模式适用于：
- 后台首页
- 预约管理
- 内容管理
- 图库管理
- 系统设置
- 后续 SEO / 安全 / 合规工作台

## 5. SEO 设计基线（后续实现必须对齐）

## 5.1 Local SEO
必须支持：
- 本地服务关键词
- 城市 / 区域关键词
- Google Maps 展示信息衔接
- NAP 一致性

### NAP 固定理解
- 店铺名称
- 地址
- 电话

## 5.2 页面 SEO
每个页面逐步支持：
- SEO Title
- Meta Description
- Canonical URL
- Open Graph
- SEO Slug

## 5.3 服务页 SEO
每个服务项目应支持独立 SEO 页面，例如：
- `/de/massage/full-body-massage`
- `/en/massage/full-body-massage`

服务详情页应逐步具备：
- 服务介绍
- 时长
- 优势说明
- FAQ
- 预约按钮
- 内链结构

## 5.4 多语言 SEO
必须继续保持 / 增强：
- 德语 / 英语独立 URL
- `hreflang`
- 多语言 sitemap

## 5.5 技术 SEO
必须继续保持 / 增强：
- SEO 友好 URL
- `sitemap.xml`
- `robots.txt`
- schema.org 结构化数据
- 图片 alt
- Core Web Vitals 基础优化

---

## 6. 安全设计基线（后续实现必须对齐）

## 6.1 后台安全
后续必须逐步具备：
- 安全登录机制
- 强密码策略
- 密码加密存储（已具基础）
- 登录失败次数限制
- 登录验证码 / 防暴力破解
- 后台会话超时
- 管理员操作日志
- 敏感操作二次确认
- 2FA 预留能力

## 6.2 预约安全 / 防恶意预约
后续必须逐步具备：
- 表单验证码 / 人机验证
- IP 频率限制
- 同一手机号 / 邮箱预约频率限制
- 黑名单机制
- 邮箱真实性校验
- 改约 / 取消安全 token
- 异常预约识别机制
- 防脚本刷预约

## 6.3 数据安全
后续必须逐步具备：
- 最少数据存储
- 后台敏感数据访问控制
- HTTPS
- 数据库备份
- 日志脱敏
- 导出限制（如未来开放）
- 数据保留与删除策略
- 防止未授权访问客户信息

## 6.4 网站基础安全
后续必须逐步具备：
- HTTPS
- CSRF 防护
- XSS 防护
- SQL 注入防护
- 文件上传校验
- API 参数校验
- 安全 Header
- 限流与基础 WAF 思路

## 6.5 通知与集成安全
后续必须逐步具备：
- SMTP 配置安全
- API 密钥管理
- Webhook 签名校验预留
- 避免日志暴露密钥或敏感信息

## 6.6 合规与隐私安全
后续必须逐步具备：
- 隐私政策勾选
- 最少必要信息采集
- 删除预约数据的基础机制
- 数据留存策略
- 后台可查看客户数据处理记录的基础能力

## 6.7 SEO 与安全平衡
必须长期遵守：
- `robots` / `sitemap` / 公开页面保持可抓取
- 后台、接口、隐私数据不可被索引
- 预约页在可用性与安全之间平衡
- 不用过度安全策略破坏公开页面 SEO

## 9. 下一阶段执行优先级（统一）

## P0：直接影响可运营后台
1. 内容页预览能力增强（边看边改）
2. 后台首页与内容页待办联动
3. 设置页继续落安全设置 / 功能开关 / 合规待办

## P1：直接影响预约安全闭环
1. token 操作更正式的日志 / 通知能力
2. 登录失败限制 / 后台基础防暴力破解
3. 预约频率限制继续增强（手机号 / 邮箱维度）
4. 隐私同意与数据留存基础机制

## P2：直接影响 SEO 与上线质量
1. 基础 SEO 信息后台化
2. Impressum / Datenschutzerklärung 页面纳入正式内容链
3. 服务详情页 SEO 深化
4. Core Web Vitals / 图片策略继续优化

## P3：部署与生产收口
1. 环境变量检查
2. 备份策略
3. 上传目录持久化与回收一致性
4. Nginx / Docker / 数据库联调

## 10. 对 AI / 后续开发者的固定要求

以后继续开发 `massage-next` 时，必须同时考虑：

- 产品设计
- 后台管理设计
- 预约流程设计
- SEO 优化
- 安全设计
- 德国本地合规基础要求

不要只做单点功能，不要只做 UI，也不要只做安全面板表达。

凡是涉及以下能力时，必须同时评估：
- 是否影响 SEO
- 是否影响安全
- 是否影响中文后台可运营性
- 是否适合单店 / 单管理员 / MVP
- 是否能低风险渐进落地
